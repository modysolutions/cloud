# Mody Custom Blocks

_Last reviewed: 2026-07-01_

This directory documents every custom Gutenberg block registered by the **Mody plugin** (`app/web/plugins/mody/`). All blocks are client-side rendered — their full HTML is written into post content by the Gutenberg `save()` function and served verbatim to the frontend. Interactive behaviour (accordion, mobile modal, keyboard navigation) is handled by `mody-frontend.js`.

---

## Block hierarchy

```
mody/sidebar          ← top-level container (place in templates/parts)
  └── mody/item       ← L1 navigation item (direct child of sidebar)
        └── mody/submenu      ← optional L2 submenu (child of L1 item)
              └── mody/item   ← L2 navigation item (child of submenu)
                    └── mody/mega-panel   ← optional L3 content panel (child of L2 item)
```

Only blocks registered as valid parents/children are accepted by the editor. The hierarchy is enforced through the `parent` and `usesContext`/`providesContext` fields in each `block.json`.

## Asset handles

All blocks share three compiled asset handles registered in `Mody\Hooks\Blocks::register_assets()`.

| Handle | File | Purpose |
|---|---|---|
| `mody-editor` | `dist/mody.js` | React edit components, loaded in the block editor only |
| `mody-style` | `dist/mody.css` | Shared styles, loaded on frontend **and** in the editor |
| `mody-frontend` | `dist/mody-frontend.js` | Vanilla-JS interactivity, loaded on the frontend only (footer) |

The compiled files are generated by the webpack pipeline:

```bash
pnpm build        # production build
pnpm start        # watch mode
```

Source lives under `src/plugins/mody/`. Each block has its own `index.js` and `.scss` file in a subdirectory, all imported by `src/plugins/mody/index.js` (the `mody-editor` entry) and `src/plugins/mody/frontend.js` (the `mody-frontend` entry).

## Rendering model

All four blocks use **static client-side rendering**:

- `save()` outputs the full HTML that is stored in the WordPress database.
- There is no active `"render"` callback — the `render.php` files in each block directory are retained as reference implementations but are not invoked.
- `useBlockProps.save()` handles block support attributes (color, spacing, custom classes) in the same way `get_block_wrapper_attributes()` previously did server-side.

### Deprecation

Each block carries a `deprecated: [{ save: () => null }]` entry so that any content saved with the old server-side-rendering approach is accepted and can be re-saved to the new static format without a validation error.

## Design tokens

Block SCSS files import `src/plugins/mody/_variables.scss` via `@use 'variables' as *`.

| Token | Value | Usage |
|---|---|---|
| `$mody-breakpoint-mobile` | `1024px` | Switches between desktop and mobile layouts |
| `$mody-sidebar-width` | `280px` | Max width of the mobile slide-in drawer |
| `$mody-transition-speed` | `0.2s` | Duration for accordion / icon transitions |
| `$mody-transition-ease` | `ease` | Easing for transitions |
| `$mody-mega-panel-z` | `100` | z-index of the mega-panel overlay |
| `$mody-modal-z` | `1000` | z-index of the mobile modal overlay |

All colour and typography values reference WordPress theme.json CSS custom properties (e.g. `var(--wp--preset--color--skobeloff)`) rather than hardcoded values.

## Adding a new block

1. Create `app/web/plugins/mody/blocks/{slug}/block.json`.
2. Add `'{slug}'` to the `BLOCKS` constant in `app/web/plugins/mody/app/Hooks/Blocks.php`.
3. Create `src/plugins/mody/{slug}/index.js` with `edit` and `save` functions.
4. Create `src/plugins/mody/{slug}.scss` for styles.
5. Import both from `src/plugins/mody/index.js`.
6. Run `pnpm build`.
7. Add a corresponding `.md` file to this directory.
