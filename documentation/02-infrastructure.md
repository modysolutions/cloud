# Infrastructure Layer

Mody Cloud runs on the Barí Docker stack. The Docker stack is defined in `compose.yml` and consists of five services communicating on a shared bridge network.

---

## Services

### nginx — Reverse Proxy

| Property | Value |
|---|---|
| Image | `nginx:alpine` |
| Ports | `80:80`, `443:443` |
| Config | `config/nginx/app.conf.template`, `config/nginx/mailpit.conf.template` |

Nginx handles all inbound HTTP/HTTPS traffic. On container startup, the official Nginx Docker image performs `envsubst` substitution on every `*.template` file in `/etc/nginx/templates/`, injecting environment variables and writing the result to `/etc/nginx/conf.d/`.

**Key behaviours:**
- HTTP → HTTPS redirect (301) for all requests
- SSL termination with certificates mounted from `config/ssl/` (generated by `./bin/certs`)
- URL rewrites route `wp-admin`, `wp-includes`, and `wp-*.php` to the isolated `/wp/` core directory
- A **production proxy fallback** for uploads: if a file under `/web/uploads/` is not found locally, Nginx transparently proxies the request to the `$PROXY_NAME` production server, so local development doesn't require a full copy of the uploads folder
- FastCGI forwarding of all `.php` requests to the PHP-FPM container on `wordpress:9000`

**Environment variables read by Nginx:**

| Variable | Purpose |
|---|---|
| `HOST_NAME` | `server_name` in both virtual hosts |
| `MAIL_HOST` | `server_name` in the Mailpit virtual host |
| `PROXY_NAME` | Production domain for uploads fallback proxy |
| `CLIENT_MAX_BODY` | `client_max_body_size` limit |

### wordpress — PHP-FPM Application Server

| Property | Value |
|---|---|
| Image | Custom built from `config/Dockerfile` |
| Base | `wordpress:{WORDPRESS_IMAGE_VERSION}-fpm-alpine` |
| Port | `9003` (Xdebug) |

The custom Dockerfile extends the official WordPress FPM Alpine image with:

- **WP-CLI** — Downloaded directly from GitHub and installed at `/usr/local/bin/wp`
- **Composer** — Installed via the official Composer installer script
- **PHP extensions** — `zip`, `gd`, `pdo_mysql`, `mysqli`, `exif`, `bcmath`
- **PECL packages** — `redis` (object cache), `xdebug` (debugging)
- **System tools** — `mariadb-client` (for `wp db` commands), `msmtp` (mail relay), `imagemagick`, `libwebp`, `libavif`
- **Custom PHP ini files** — Copied from `config/php/` at build time

**PHP configuration files:**

| File | Purpose |
|---|---|
| `config/php/uploads.ini` | Upload size and execution time limits |
| `config/php/sendmail.ini` | Routes PHP `mail()` through msmtp |
| `config/php/zzz-mail.ini` | Forces msmtp as the sendmail binary |
| `config/php/xdebug.ini` | Xdebug mode, port, and log path |

**WordPress constants injected via `WORDPRESS_CONFIG_EXTRA`:**

The Docker Compose environment block passes a multi-line PHP string as `WORDPRESS_CONFIG_EXTRA`. The official WordPress Docker image appends this string to `wp-config.php` on first start. Constants set this way include:

- `WP_MEMORY_LIMIT` — PHP memory available to WordPress
- `WP_ENV` / `WP_ENVIRONMENT_TYPE` — Environment name (`local`, `stage`, `production`)
- `WP_AUTO_UPDATE_CORE` — Automatic core update control
- `WP_POST_REVISIONS` — Number of post revisions to keep
- `WP_DEBUG`, `WP_DEBUG_LOG`, `WP_DEBUG_DISPLAY` — Debug output control
- `FORCE_SSL_ADMIN` — Force HTTPS in WP admin
- `DISALLOW_FILE_EDIT` — Disable in-browser file editing
- SMTP constants (`WPMS_*`) — Configuration for WP Mail SMTP plugin
- `DISABLE_WP_CRON` — Disable the pseudo-cron (use real cron instead)
- `COMPRESS_CSS`, `COMPRESS_SCRIPTS`, `ENFORCE_GZIP` — Performance flags

### wp_db — MariaDB Database

| Property | Value |
|---|---|
| Image | `mariadb:${MARIADB_VERSION}` (see `.env`) |
| Port | `3306:3306` |
| Data | Named volume `{CONTAINER_PREFIX}data` |

MariaDB stores all WordPress data. The `wordpress` service depends on `wp_db` with `condition: service_healthy` — Docker Compose will not start the PHP container until the database passes its healthcheck.

**Healthcheck:** `healthcheck.sh --su-mysql --connect --innodb_initialized`

Custom MariaDB configuration is mounted from `config/my.cnf`.

---

### redis — Object Cache

| Property | Value |
|---|---|
| Image | `redis:alpine` |
| Healthcheck | `redis-cli ping` |

Redis is available in the network for WordPress object caching. The PHP container has the `redis` PECL extension installed. A Redis object cache plugin (e.g. WP Redis or Redis Object Cache) must be activated separately to use it.

---

### mailpit — Email Trap

| Property | Value |
|---|---|
| Image | `axllent/mailpit` |
| Data | Named volume `{CONTAINER_PREFIX}mail_data` |

Mailpit intercepts all outgoing email sent by WordPress via SMTP, preventing accidental email delivery to real addresses during development. The web UI is accessible at `https://$MAIL_HOST` (proxied through Nginx).

**Configuration:**
- `MP_MAX_MESSAGES: 5000` — Maximum messages stored
- `MP_SMTP_AUTH_ACCEPT_ANY: 1` — Accept any SMTP credentials
- `MP_SMTP_AUTH_ALLOW_INSECURE: 1` — Allow insecure SMTP connections

PHP sends mail via msmtp, which is configured to forward to `mailpit:1025`.

## Volumes

| Volume | Name | Purpose |
|---|---|---|
| `wp_data` | `{CONTAINER_PREFIX}data` | MariaDB data persistence |
| `mail_data` | `{CONTAINER_PREFIX}mail_data` | Mailpit message storage |

---

## Network

All services share a single bridge network named `{CONTAINER_PREFIX}network`. Container-to-container communication uses service names as hostnames (e.g. `wordpress:9000`, `mailpit:1025`, `wp_db:3306`).

---

## Log Directories

| Path on host | Mounted to | Used by |
|---|---|---|
| `./app/logs/` | `/var/www/html/logs/` | Xdebug log, WordPress debug log |
| `./logs/nginx/` | `/var/log/` (nginx, read-only) | Nginx access and error logs |
| `./logs/wordpress/` | `/var/log/` (wordpress, read-only) | PHP-FPM logs |

> **Note:** `./app/logs/` is created by `./bin/install`. The `./logs/nginx/` and `./logs/wordpress/` directories are also created by `./bin/install` but Docker Compose will create them automatically (as root) if they don't exist.
