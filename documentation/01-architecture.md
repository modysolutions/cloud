# Mody Cloud — Project Architecture

## Overview

Mody Cloud is a Barí-powered WordPress hosting and developer platform. This repository is the Mody Cloud project itself, built on top of the **[Barí WordPress framework](https://github.com/bariwp/bari)** — a self-contained Docker-based WordPress stack designed for *digital sovereignty*: providing teams with a fully controlled, reproducible environment free from dependency on centralised SaaS services.

Mody Cloud is a founding sponsor of Barí and uses Barí as its canonical project runtime. The repository is structured as a **monorepo** with clean separation between infrastructure configuration, automation tooling, frontend source code, and the WordPress application.

## Key Architectural Decisions

These decisions are inherited from the Barí framework and form the technical foundation of the Mody Cloud project.

### 1. WordPress Core Isolation (Bedrock-like)
WordPress core files live in `app/wp/` rather than at the web root. Nginx rewrites `wp-admin`, `wp-includes`, and `wp-*.php` paths to `/wp/`, keeping core files separate from application code in `app/web/`. This makes upgrades safer and version control cleaner.

### 2. Composer-Managed Dependencies
All third-party plugins and themes are installed and updated via Composer using [WPackagist](https://wpackagist.org) as the package repository. No manual plugin downloads. Premium plugins (e.g. ACF Pro) can be served via a private [Satispress](https://www.satispress.com) instance.

### 3. Environment-Driven Configuration
Every runtime constant — database credentials, WordPress settings, SMTP, memory limits, debug flags — is injected via environment variables from `.env`. The Docker image reads these and writes `wp-config.php` automatically. There is no hardcoded configuration in the codebase.

### 4. Gutenberg-Native Theme Rendering
The custom theme in `app/web/themes/theme/` is Gutenberg-native. WordPress block templates in `templates/*.html`, template parts in `parts/*.html`, and `theme.json` own the frontend page shell and editor design system. PHP hook classes handle theme support, assets, cleanup, and compatibility wiring.

[Timber](https://timber.github.io/docs/) and [Twig](https://twig.symfony.com/) are installed through Composer, but they are not the primary theme rendering model. Use Timber/Twig only for compatibility code or for plugins that need explicit server-side-rendered views.

### 5. Container-First CLI
All day-to-day commands (Composer, WP-CLI) are proxied through shell scripts in `bin/` that transparently route to the correct Docker container. Developers never need to type raw `docker exec` commands.
